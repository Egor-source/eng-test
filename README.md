# Аутентификация

В текущем репозитории используется **JWT (JSON Web Token)**. Несмотря на популярность, у этой технологии есть свои особенности.

## Альтернативные техники

В проектах с высокими требованиями к безопасности (например, банковские приложения) я использовал аутентификацию на базе сессий (**stateful sessions**) с хранением в Redis. Это позволяет мгновенно отзывать доступ у пользователя, что в случае с чистым JWT сделать сложно без введения дополнительных механизмов (Blacklist).

## Выбор JWT

Если бы у меня был выбор, я бы все равно предпочел JWT для этого проекта (браузерная игра), так как он позволяет легко масштабировать бекенд (**stateless подход**). Это критично при запуске нескольких инстансов приложения, как указано в требованиях.

## Когда выбрать другое

В системах с очень строгим контролем сессий или там, где объем передаваемых данных в каждом запросе критичен (например, для мобильных сетей с плохой связью), я бы выбрал **opaque tokens**, хранящиеся в БД/Redis, чтобы минимизировать размер заголовков запроса.

# Идентификация

Для идентификации записей в таблицах проекта используется **UUID**.

## Преимущества

Это позволяет генерировать идентификаторы на стороне клиента или в распределенных системах без риска коллизий и без обращения к центральной БД за следующим ID.

## Проблемы UUID

В моей практике UUID становился проблемой в высоконагруженных БД с огромным количеством записей (десятки миллионов). Использование UUID (особенно версии 4) в качестве первичного ключа может приводить к сильной фрагментации индексов в B-Tree, так как новые записи вставляются в случайные места, что замедляет запись.

В таких случаях я бы предпочел **ULID** или **UUID v7**, которые сохраняют уникальность, но при этом сортируемы по времени.

# Организация кода

Серверная часть построена на **NestJS**, используя DI, контроллеры и сервисы.

## Польза механизмов

### Dependency Injection (DI)

Лично для меня это упростило unit-тестирование. Можно легко подменять реальные сервисы (например, сервис для работы с БД) на моки при тестировании логики.

### Декораторы

Использование декораторов для валидации и определения прав доступа позволяет держать контроллеры чистыми, вынося инфраструктурный код за пределы бизнес-логики.

## Сложности

Чрезмерное увлечение middleware и сложными интерцепторами иногда создавало "магию", которую сложно отлаживать новым разработчикам в команде. Я стараюсь ограничивать их использование только общими задачами (логирование, авторизация).

# Реактивность

Проект использует **React** для интерфейса.

## Проблемы реактивности

Реактивность становилась помехой при реализации сложных графических элементов или канваса, где частота обновлений очень высока. В одном из проектов я выносил логику отрисовки части интерфейса из-под контроля React в прямой манипулятор DOM, чтобы избежать лишних перерисовок всего дерева компонентов.

## Неудобства в React

### Сложность useEffect

Работа с зависимостями часто приводит к бесконечным ререндерам или использованию устаревших замыканий (**stale closures**), что требует высокой концентрации при разработке.

### Отсутствие встроенного стейт-менеджера

Для синхронизации состояния раунда и очков между страницами часто приходится подключать сторонние библиотеки (**Zustand**, **Redux**), что усложняет проект.
